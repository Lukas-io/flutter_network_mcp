# 0.9.0 WebSocket frame capture, SHELVED (do not merge)

> **Status: built, verified by unit tests, and parked.** The code on
> `feat/0.9.0-websocket-capture` works, but it ships an in-app companion
> package, and the maintainer has decided **not** to require any code inside
> the host app. So this line does not merge as-is. It stays on the branch as a
> reference in case a no-app-code approach ever becomes possible.

## What was built

- `flutter_network_mcp_hooks/` companion package: intercepts dart:io
  WebSockets via `HttpOverrides` + `detachSocket()`, decodes RFC 6455 frames,
  reassembles fragments, inflates permessage-deflate (RFC 7692), and buffers
  one row per logical message. Drains over the
  `ext.flutter_network_mcp.getRealtimeProfile` VM service extension. 15 tests.
- MCP side: schema v8 (`websocket_connections` + `websocket_frames` tables, a
  v7->v8 migration), a realtime poll in the capture writer, a `realtime`
  capability category, and `ws_list` / `ws_get` tools. 6 storage tests; full
  suite green; `dart analyze` clean.

## Why it is shelved

The feature is correct, but its **only viable delivery mechanism requires code
inside the host app**, and that is a hard no for this maintainer.

The chain of reasoning (do not re-litigate without new facts):

1. **The Dart VM service exposes no WebSocket frame data.** Confirmed by
   grepping every `vm_service` 14.2.1 through 15.2.0: no `getWebSocketProfile`,
   no frame stream, nothing. See `docs/0.8.0-realtime-investigation.md`.
   `getHttpProfile` sees only the upgrade *handshake* (an HTTP GET).
   `getSocketProfile` sees only byte *counters* (`readBytes` / `writeBytes`),
   never payloads. After dart:io calls `detachSocket()` to take over framing,
   the VM service is blind to every frame. So a passive, VM-service-only
   observer cannot work.

2. **A parallel "shadow" connection cannot observe the app's traffic.** A
   WebSocket is point-to-point, not pub/sub. A second connection to the same
   URL is a separate session: it receives only what the server sends *to it*,
   never a copy of the app's inbound stream, and never the app's *outbound*
   messages at all. With Socket.IO (per-connection `sid`, JWT/room auth, e.g.
   sanga's `/events` `/orders` `/driver` gateways) the second connection is a
   distinct authenticated session. Worse, replaying the app's auth could cause
   real side effects (place an order, send a message) and, on single-session
   servers, kick the app offline. Rejected on both feasibility and safety.

3. **In-app hooks (this branch) or a MitM proxy are the only ways to see
   frames.** The proxy path also requires app/device changes (route through
   it + install a CA cert) and breaks cert pinning, so it is strictly worse,
   not a no-app-code alternative. Hooks were rejected by the maintainer
   ("never input code into the user's app"). With both off the table, there is
   no shippable WebSocket *frame* capture for this tool.

## What still works without any of this

WebSocket *connections* remain visible for free, no companion needed:
`socket_list` shows the upgraded socket's byte counters and open/closed state,
and `network_list` shows the upgrade handshake request. Only the frame
*contents* are out of reach.

## To un-shelve

Only if one of these changes: the Dart SDK adds WebSocket profiling to the VM
service (revisit, drop the companion entirely), or a future requirement makes
an opt-in in-app dependency acceptable to the maintainer. Until then, this
branch is a parked reference, not a release.
