# 0.8.0 investigation — WebSocket + gRPC frame capture

> **Status: investigation outcome = option 3** (architecture proposal,
> no working feature in this PR). The original 0.8.0 plan was
> "investigation-first" with three accepted outcomes; this PR documents
> findings + recommends a path forward, but ships NO frame-capture code.
> No version bump. The actual feature ships in a future release once
> the recommended architecture lands.

## What we wanted

From the 0.7.x roadmap top-10 list (`#10 — WebSocket + gRPC frame capture`):

- **WebSocket**: per-frame capture (text/binary, direction, timestamp) persisted to a new `websocket_frames` table; tools `ws_list` / `ws_get` for inspection.
- **gRPC**: HTTP/2 framing decoded; messages persisted as base64-encoded binary (+ schema-decoded text when a `.proto` is registered); tools `grpc_list` / `grpc_get`.
- Both under a new `realtime` capability category. Schema bumps to v8.

## What we found

**VM service exposes nothing usable for either.** The investigation agent grep'd every version of `package:vm_service` from 14.2.1 through 15.2.0:

| Capability | API surface | Verdict |
|---|---|---|
| `getWebSocketProfile`, `enableWebSocketProfiling`, `WebSocketProfile`, etc. | **NONE** | NOT FOUND |
| `getHttp2`, `Http2Profile`, `Http2Stream` | **NONE** | NOT FOUND |
| `grpc`, `protobuf` mentions | **NONE** | NOT FOUND |
| `SocketStatistic` payload field | Only `readBytes` + `writeBytes` counters at `dart_io_extensions.dart:214–262` | NO frame data |
| VM service named streams (`HTTP_TIMELINE`, etc.) | `kExtension`, `kTimeline`, `kTimelineEvents` exist — none expose realtime HTTP/WS data | NO traffic stream |
| Debug-mode introspection hooks | **NONE** for WebSocket upgrade or HTTP/2 stream framing | NOT FOUND |

The existing `getHttpProfile` (which `flutter_network_mcp` already uses) is **HTTP/1.1 via `dart:io HttpClient` ONLY**. It does not see:

- WebSocket frames (handshake passes through `HttpClient` and IS captured, but everything after the upgrade is invisible at the VM service layer).
- HTTP/2 streams (gRPC apps use `package:grpc` which has its own HTTP/2 stack — bypasses `dart:io HttpClient`).
- gRPC messages.

Socket bytes show up in `getSocketProfile`, but only as **aggregate counters** (`read_bytes`, `write_bytes`). No payload. No framing.

**Conclusion: the current vm_service-based architecture cannot deliver the 0.8.0 plan as designed.** We need a different approach.

## Three architectural alternatives

### Option A — Companion Dart package (recommended)

**Sketch.** Ship a companion package `flutter_network_mcp_hooks` (published to pub.dev separately). User adds it as a `dev_dependencies` entry + a one-line install:

```dart
// In the user's main.dart, top of main():
import 'package:flutter_network_mcp_hooks/flutter_network_mcp_hooks.dart';

void main() {
  if (kDebugMode) FlutterNetworkMcpHooks.install();
  runApp(...);
}
```

`install()` wraps the runtime to capture what vm_service can't:

- **WebSocket**: shadow `WebSocket.connect` via `IOOverrides`; wrap the returned `WebSocket` so every `add` / listener event records a frame.
- **HTTP/2 / gRPC**: hook `package:http2`'s transport-level callbacks (or `package:grpc`'s channel-level hooks if exposed) to record streams + messages.
- **Raw sockets** (future): hook `Socket.connect` for protocols neither side covers.

Events go to an in-memory ring buffer in the user's app. A new VM service extension method (`ext.flutter_network_mcp.getRealtimeProfile`) exposes drain + clear, mirroring how `getHttpProfile` works today. The MCP polls it every 2s alongside the existing HTTP poll. New schema tables: `websocket_frames`, `grpc_messages`.

**Pros:**
- Agent-first model preserved — user only edits ONE file in their app.
- Captures everything (WS, HTTP/2, future protocols).
- Consistent with how DevTools extends app introspection.
- No proxy infrastructure for the user to maintain.
- No TLS / MitM complications.

**Cons:**
- Requires the user to add a dependency + one-liner. The "zero-config" story degrades for realtime traffic specifically (HTTP/1.1 stays zero-config).
- Each new protocol needs hooks code.
- Companion package needs its own release + maintenance cycle.

### Option B — Local proxy mode

**Sketch.** The MCP starts a local HTTP/HTTPS/WS proxy server. User configures their Flutter app to route traffic through it via `HttpOverrides` or system proxy settings. The proxy MitMs TLS using a generated CA cert the user installs.

**Pros:**
- Captures EVERYTHING regardless of protocol or library.
- No user-side dependency on a companion package.
- Could even capture native HTTP calls outside `dart:io`.

**Cons:**
- Heavy user setup (proxy config + CA cert install + accept-cert prompts on devices).
- TLS MitM is a security footgun.
- Significantly larger codebase (proxy server + cert management).
- Mobile apps frequently pin certs; proxy breaks them silently.
- Doesn't fit the agent-first model — agent can't deploy a proxy to the user's phone.

### Option C — Wait for VM service

**Sketch.** Open an issue against `dart-lang/sdk` requesting WebSocket profiling parity with `getHttpProfile`. Wait.

**Pros:**
- Zero code on our side.
- If accepted, becomes the canonical solution.

**Cons:**
- Multi-quarter timeline at best, never at worst.
- Doesn't address HTTP/2 / gRPC at all (those are out-of-tree packages).

## Recommendation

**Option A.** It's the only path that preserves the agent-first / zero-setup model for HTTP/1.1 while adding realtime coverage in a controlled way. The companion-package adoption cost (one dev_dependency + one line) is the same cost the user already pays for DevTools features they value. Option B's TLS-MitM-on-mobile failure mode would generate more bug reports than the feature itself would resolve.

## Suggested phasing for the companion package (next release line)

| Release | Scope |
|---|---|
| 0.9.0 | `flutter_network_mcp_hooks` v0.1 published to pub.dev. WebSocket capture only. MCP schema v8 + `ws_list` / `ws_get` tools. README explains the install step. |
| 0.9.1 | gRPC via `package:grpc` channel hooks. `grpc_list` / `grpc_get` tools. Schema v9. |
| 0.9.2 | HTTP/2 capture for non-gRPC HTTP/2 traffic via `package:http2` direct hooks. |
| 1.0.0 | Full realtime category stable; opt-out and capability gating polished. |

## What this PR (0.8.0) does ship

Just this doc. No version bump. No code. The PR exists so that:

1. The investigation findings are captured in the repo, not just in a planning conversation.
2. Future maintainers (or future-you in three months) can see WHY 0.8.0 didn't ship the original plan, and which architectural decision was made.
3. The recommendation gives the next planning round a starting point instead of a blank slate.

The corresponding `FUTURE_FEATURES.md` entry has been updated to reflect this — the WS/gRPC item moves out of "honorable mention" and into "next-release-line architecture proposal."
